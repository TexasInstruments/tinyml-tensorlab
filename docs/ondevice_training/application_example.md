## 1. Overview

This is a complete working example that demonstrates on-device training for anomaly detection on the F29H85x microcontroller. The application trains an autoencoder model directly on the device using embedded data, recalculates the anomaly detection threshold, and evaluates the model — all without any PC involvement after deployment.

**What this example demonstrates:**
1. Golden vector validation (verify artifacts are loaded correctly)
2. Pre-training inference (baseline — shows the model hasn't learned yet despite coincidental accuracy)
3. On-device training with early stopping (model learns from scratch or fine-tunes)
4. Automatic threshold recalculation using validation data
5. Post-training inference 

**Result from the example run:** The model trains from completely untrained weights (0 PC-side epochs). While pre-training accuracy happened to be 100% on this small test set by coincidence (the initial threshold landed in the right spot), the model had not actually learned anything — normal reconstruction errors were ~5800. After on-device training, normal errors dropped to ~10 and the separation between normal and anomaly improved from 1.18× to 8.6×, producing a genuinely robust detector.

### Prerequisites

- **Hardware:** F29H85x LaunchPad
- **Software:** F29H85x SDK, Code Composer Studio (CCS), C29 CGT Tools
- **Artifacts generated** via ModelZoo with `ondevice_training: True` (see [Running ModelZoo for ODT](running_modelzoo_for_odt.md))
- **Conceptual background:** [Overview](overview.md)

### Project location

### Stage 2: Golden Vector Validation

A known test input is processed through the complete pipeline (feature extraction → frozen model → trainable model) and the output is compared against a pre-computed golden reference.

    // Feature extraction
    FE_initFeatureExtract(fe_handle);
    FE_runFeatureExtract(fe_handle);
    
    // Frozen model
    tvmgen_default_run(&inputs, &outputs);
    
    // Trainable model forward pass
    AnomalyDetection_ForwardPass(&ad_model_ctx, frozen_output, model_input, all_model_output, &reconstruction_error);
    
    // Compare against golden output
    for (output_index = 0; output_index < FE_NN_OUT_SIZE; output_index++) {
        float difference = fabs(golden_output[output_index] - all_model_output[output_index]);
        if (difference > 1e-3) error++;
    }

**Console output:**

    Golden vectors matched: 48 not matched: 0

All 48 output elements match within tolerance, confirming that `mod.a`, `trainable_model_config`, and feature extraction are all loaded and working correctly.

**Note:** The 100% pre-training accuracy is **misleading** — the model has not been trained at all (0 PC-side epochs). The autoencoder has not learned to reconstruct normal data, which is evident from the extremely high reconstruction errors: normal samples produce errors around 5700–5900, meaning the reconstruction is essentially garbage. The tiny margin between normal errors (~5900) and the threshold (~6430) means even slight data variation could flip predictions. This is **not** a usable model — on-device training is needed to actually learn meaningful reconstructions.

### Stage 4: On-Device Training

The application waits for the user to trigger training via the CCS debugger.

    printf("Set start_training=1 in debugger to begin training...\n");
    while (!start_training) {}

**How to trigger:** In CCS, open the **Variables** view , add `start_training`, and change its value from `0` to `1`.

Once triggered, the training loop runs:

    AnomalyDetection_StartTraining(&ad_model_ctx);
    
    while (current_mode == MODE_TRAINING) {
        if (ad_model_ctx.current_phase == PHASE_TRAIN) {
            process_one_sample(&train_iter);     // Feed training data
        } else {
            process_one_sample(&val_iter);       // Feed validation data
        }
        AnomalyDetection_ProcessTrainingFrame(&ad_model_ctx, frozen_output, model_input);
        AnomalyDetection_GetMode(&ad_model_ctx, &current_mode);
    }

The library automatically:
- Alternates between training and validation phases each epoch
- Applies EMA smoothing to validation loss
- Saves best weights when validation improves
- Triggers early stopping after `patience` epochs without improvement

### Stage 5: Threshold Recalculation

After training, the model's reconstruction errors are very different, so the threshold must be recalculated. The library automatically transitioned to `MODE_THRESHOLD_CALC` after training stopped.

After training, the model has genuinely learned what "normal" looks like: normal errors dropped to ~10, while anomaly errors remain high (~130–1000). The **separation ratio** improved from 1.18× to 8.6×, meaning the detector is now robust to noise and threshold sensitivity rather than relying on a lucky threshold placement.

## 4. Key Code Patterns

### 4.1 process_one_sample()

This function processes one raw sample from the dataset through feature extraction and the frozen model, producing `frozen_output` ready for the trainable model.

    void process_one_sample(DatasetIterator_t* iter) {
        // For each frame in num_frame_concat:
        //   1. Extract raw sensor data for all channels from the sample
        //   2. Shift feature history (for temporal context)
        //   3. Run feature extraction (FE_runFeatureExtract)
        
        // After all frames processed:
        //   4. Run frozen model (tvmgen_default_run)
        //   5. Advance iterator (wraps around at end)
    }

**Why this complexity?** Raw sensor data has multiple channels (X, Y, Z) and potentially multiple frames for temporal context. The feature extraction pipeline processes one frame at a time, maintaining a history buffer for concatenation. This function encapsulates that multi-frame, multi-channel processing.

### 4.2 DatasetIterator_t

A simple struct for cycling through embedded datasets:

    typedef struct {
        const float (*inputs)[RAW_INPUT_SIZE];   // Pointer to 2D array
        const int16_t* labels;                    // Pointer to labels
        uint16_t num_samples;                     // Total samples
        uint16_t current_index;                   // Current position (wraps)
    } DatasetIterator_t;

Three iterators are used — `train_iter`, `val_iter`, `test_iter` — each pointing to the corresponding arrays in `ondevice_training_data.c`.

### 4.3 Training Loop Pattern

The training loop checks the library's internal phase to decide which data to feed:

    while (current_mode == MODE_TRAINING) {
        if (ad_model_ctx.current_phase == PHASE_TRAIN) {
            process_one_sample(&train_iter);      // Training data
        } else {
            process_one_sample(&val_iter);        // Validation data
        }
        AnomalyDetection_ProcessTrainingFrame(&ad_model_ctx, frozen_output, model_input);
        AnomalyDetection_GetMode(&ad_model_ctx, &current_mode);
    }

The library handles everything internally — epoch counting, phase switching, batch management, early stopping. The application just feeds the right data and checks when training is done.

### 4.4 Debugger Variable Trigger

    volatile uint16_t start_training = 0;
    // ...
    while (!start_training) {}

The `volatile` keyword prevents the compiler from optimizing away the loop. In CCS, use **Variables**  view to change `start_training` to `1` while the application is running.

## 5. Hyperparameter Reference

| Parameter | Value | Set In | What It Controls |
|-----------|-------|--------|-----------------|
| `learning_rate` | 0.0001 | `main()` | Weight update step size |
| `patience` | 10 | `main()` | Epochs without improvement before early stop |
| `min_improvement_delta` | 0.5 | `main()` | Minimum loss decrease to count as improvement |
| `ema_alpha` | 0.5 | `main()` | Validation loss smoothing (higher = more reactive) |
| `batches_per_epoch` | 10 | `main()` | Training samples per epoch (× batch_size) |
| `threshold_samples` | 500 | `main()` | Normal samples for threshold calculation |
| `threshold_percentile` | 0.99 | `main()` | Percentile for threshold (if percentile method) |
| `gaussian_std_multiplier` | 4 | `main()` | Std devs above mean (if Gaussian method) |
| `use_percentile_threshold` | 1 | `main()` | 1 = percentile, 0 = Gaussian |
| `inference_window_size` | 1 | `main()` | Majority voting window (1 = no voting) |
| `stats_window_size` | 100 | `main()` | Frames per statistics report |
| `initial_threshold` | 6430.98 | `user_input_config.h` | Pre-training threshold from ModelMaker |
| `TRAIN_BATCH_SIZE` | 1 | `trainable_model_config.h` | Samples per weight update (compile-time) |
| `VAL_BATCH_SIZE` | 1 | `trainable_model_config.h` | Samples per validation evaluation (compile-time) |

## 6. How to Adapt for Your Own Data

To use this example with a different dataset or sensor:

1. **Generate new artifacts** via ModelZoo with your dataset and YAML configuration (see [Running ModelZoo for ODT](running_modelzoo_for_odt.md))

2. **Replace artifact files** in the `artifacts/` directory:
   - `mod.a` — new frozen model
   - `tvmgen_default.h` — new frozen model API
   - `trainable_model_config.h/.c` — new trainable architecture and weights
   - `ondevice_training_data.h/.c` — new embedded dataset
   - `user_input_config.h` — new feature extraction config
   - `test_vector.c` — new golden vectors

3. **Adjust hyperparameters** in `main()` if needed:
   - Different data may need different `learning_rate` (start with 0.0001, adjust if training diverges or converges too slowly)
   - Adjust `patience` based on expected convergence time
   - Adjust `threshold_samples` based on available validation data

4. **Modify `process_one_sample()`** if your sensor pipeline differs (different number of channels, different frame structure, etc.)

5. **Rebuild** the CCS project and flash the device
